## 1. What This System Does

This is a **fully automated stock watchlist intelligence pipeline** that:

- Runs **twice a week** (Monday = owned portfolio, Thursday = watchlist candidates) via GitHub Actions — no server needed, completely free to run
- **Fetches live data** for every ticker: price, fundamentals, technicals, sentiment from Yahoo Finance (free), SEC EDGAR (free), FRED (free), Finnhub (free tier), NewsAPI (free tier)
- **Scores each stock 0–10** across 11 categories: Fundamentals, Valuation, Technical, Sentiment, Management, Moat, Partnerships, Macro, Catalysts, Risk, Institutional
- **Detects signal changes** (CONFLUENCE / CONSOLIDATION / SQUEEZE ON / RISK WATCH) and alerts you when a stock "flips" signal — the key action trigger
- **Sends a weekly email** with a TLDR digest, signal flips, and analyst upside data
- **Publishes a full HTML report** to GitHub Pages (your permanent link) with score gauges, radar charts, category breakdowns, bull/bear flags, and thesis
- **Hosts an interactive Streamlit dashboard** (live, on-demand analysis for any ticker, portfolio P&L tracker, score trend charts, analyst consensus)
- **Optionally sends Telegram alerts** when a flip or strong entry signal is detected intraday

**Everything runs on free tiers.** The only optional paid service is Claude AI for thesis text (~$0.45/month) and Quiver Quant for dark pool data ($30/month, optional).

## 5. Step-by-Step Rebuild

If starting from scratch on a new account or fresh machine:

### Step 1 — Create the GitHub repository

1. Go to [github.com](https://github.com) → New repository
2. Name it `stock-watchlist-pipeline` (or anything — update the Pages URL in `watchlist.yml` if you change it)
3. Set to **Public** (required for free GitHub Pages)
4. Don't add any files — you'll push everything

### Step 2 — Clone and push the code

```bash
git clone https://github.com/YOUR_USERNAME/stock-watchlist-pipeline.git
cd stock-watchlist-pipeline

# Copy all project files here, then:
git add .
git commit -m "Initial pipeline setup"
git push origin main
```

### Step 3 — Enable GitHub Pages

1. Repository → **Settings** → **Pages**
2. Source: **GitHub Actions** (not "Deploy from branch")
3. Save. The Pages URL will be `https://YOUR_USERNAME.github.io/stock-watchlist-pipeline`

### Step 4 — Set GitHub Actions secrets

Repository → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**

Add each one (see table in Section 4). The minimum to get started:
- `GMAIL_USER`
- `GMAIL_APP_PASSWORD`
- `NOTIFY_EMAIL`

Everything else is optional and can be added later.

### Step 5 — Enable GitHub Actions write permission

Repository → **Settings** → **Actions** → **General** → scroll to **Workflow permissions**
→ Select **"Read and write permissions"** → Save

This is required because the pipeline commits history CSVs back to the repo.

### Step 6 — Customize your tickers

Edit `config/owned.txt` — add your portfolio tickers (one per line):
```
AAPL
NVDA  mega
CRDO  smallg
```

Edit `config/watchlist.txt` — add research candidates:
```
AMD   largeg
PLTR  largeg
```

Optionally edit `config/thesis_scores.yaml` to add your conviction scores (see Section 6).

### Step 7 — Run the pipeline manually

GitHub → **Actions** → **Weekly watchlist digest** → **Run workflow** → choose group (`owned` or `watchlist`) → **Run workflow**

The pipeline takes 5–8 minutes. When done:
- Check your email for the digest
- Visit your GitHub Pages URL for the full report
- History CSVs are committed back to `data/history/`

### Step 8 — Deploy Streamlit

1. Go to [share.streamlit.io](https://share.streamlit.io)
2. Sign in with GitHub
3. New app → select your repo → branch `main` → Main file: `streamlit_app.py`
4. Deploy
5. After deploy, go to app settings → **Secrets** → add your API keys (TOML format, see Section 4)

### Step 9 — Verify everything works

- [ ] Pipeline ran without errors (check Actions tab)
- [ ] Email received with report link
- [ ] GitHub Pages loads the HTML report
- [ ] Streamlit dashboard loads without errors
- [ ] Analyze tab: type `AAPL`, click Analyze — should show live data
- [ ] Home tab: Score Trend section loads (requires at least one pipeline run to create `data/history/*.csv`)

## 6. Configuration Files Explained

### `config/owned.txt`

One ticker per line. Optional archetype after a space.

```
# Archetype options: mega  largeg  smallg  spec  micro
# If omitted, archetype is auto-detected from market cap

AAPL        # auto-detect (will be "mega")
NVDA  mega  # explicit
CRDO  smallg
```

**Archetype matters** because scoring thresholds scale with company size. A 20% revenue growth is impressive for a mega-cap but unremarkable for a spec-cap.

| Archetype | Market Cap | Rev Growth "green" threshold |
|-----------|-----------|------------------------------|
| `mega` | >$200B | >12% |
| `largeg` | $10B–$200B | >12% |
| `smallg` | $1B–$10B | >20% |
| `spec` | $300M–$1B | >30% |
| `micro` | <$300M | >30% |

### `config/watchlist.txt`

Same format as owned.txt. Runs on Thursday. Any ticker that appears in both is automatically skipped on Thursday (no double-processing).

### `config/thesis_scores.yaml`

Your stable conviction layer — updated **monthly** after earnings or major news. This prevents the pipeline from whipsawing on daily price moves.

```yaml
NVDA:
  last_updated: "2026-04-30"
  fundamentals: 9        # 0–10, or null to auto-score
  valuation: 6
  moat: 9
  management: 8
  partnerships: 8
  macro: 7
  risk: 7
  overall_conviction: 9
  thesis_one_liner: "AI infrastructure monopoly with pricing power and a decade of secular demand ahead."
```

Categories you can manually score:
- `fundamentals` — revenue growth, FCF, gross margin, debt
- `valuation` — PE, PEG, EV/EBITDA, analyst target
- `moat` — network effects, switching costs, IP, brand
- `management` — track record, insider buying, guidance accuracy
- `partnerships` — strategic deals, channel, ecosystem
- `macro` — TAM growth, rates sensitivity, regulatory tailwinds/headwinds
- `risk` — execution risk, debt, competition, key-person

Leave any category as `null` and the scorer auto-fills it from live data.

## 7. Source Files Explained

### `src/fetcher.py`

Pulls all data for a ticker. Returns a `dict` with ~50+ fields including:
- `price`, `price_change_1d_pct` — from yfinance
- `revenue_growth_yoy`, `gross_margin`, `operating_margin`, `free_cashflow`, `pe_forward` — from yfinance info
- `edgar_gross_margin`, `edgar_fcf` — fallback from SEC EDGAR if yfinance returns None
- `rsi_14`, `sma_50`, `sma_200`, `vs_50`, `vs_200` — computed from price history
- `inst_ownership_pct`, `short_float_pct` — institutional + short data
- `analyst_target`, `analyst_upside`, `recommendation` — analyst consensus
- `fh_eps_surprise_pct`, `fh_beat`, `fh_consecutive_beats` — from Finnhub
- `fh_insider_net_shares_90d`, `fh_insider_bullish` — from Finnhub
- `earnings_date` — next earnings

### `src/scorer.py`

Archetype-aware scoring engine. Returns a `ScoreResult` with:
- `weighted_score` — 0–10 overall weighted score
- `weighted_light` — "green" (≥7), "yellow" (5–6.9), "red" (<5), "gray" (no data)
- `categories` — dict of `CategoryResult` objects, each with `.score`, `.light`, `.flags_bull`, `.flags_bear`
- `bull_flags` / `bear_flags` — top signals across all categories
- `archetype` — the archetype used for threshold calibration

The weighting per category (these are the defaults; thesis_scores.yaml overrides individual scores):

| Category | Weight |
|----------|--------|
| Fundamentals | 20% |
| Valuation | 15% |
| Technical | 15% |
| Sentiment | 10% |
| Management | 8% |
| Moat | 8% |
| Macro | 7% |
| Catalysts | 7% |
| Risk | 5% |
| Partnerships | 3% |
| Institutional | 2% |

### `src/signals.py`

Classifies each ticker into one of four signals based on score + technicals:

| Signal | Meaning | Typical Setup |
|--------|---------|--------------|
| `CONFLUENCE` | Thesis + technicals aligned | High score, RSI 45–70, above SMA-50 |
| `CONSOLIDATION` | Strong thesis, rangebound | High score, tight price action |
| `SQUEEZE ON` | Good thesis, technical pressure | High score but RSI<35 or below SMA-200 |
| `RISK WATCH` | Deteriorating signal | Score declining or key threshold broken |

Also detects **flips** (signal changed since last run) and **score trend** (up/flat/down). Flips trigger email highlights and Telegram alerts.

### `src/report.py`

Generates `public/index.html` — the full GitHub Pages report. Key features:
- Dark navy design (`#0f172a` background, `#1e293b` cards)
- **SVG gauge** — pure SVG semicircle speedometer, 0–10 scale, green/yellow/red zones, no external JS
- **SVG radar chart** — hexagonal spider chart for 6 weighted factors
- **Analyst consensus block** — recommendation label + spectrum gradient bar + target price + upside %
- **Category breakdown** — 11 expandable categories with per-category bull/bear flags
- **Metrics strip** — horizontal scrolling key metrics with semantic color labels
- **Macro panel** — Fed Funds Rate, 10Y yield, CPI, yield curve regime
- **Accuracy scorecard** — 60-day signal hit rate vs SPY+2%
- **Sector radar** — compact grid of research watchlist peers

### `src/emailer.py`

Sends HTML email via Gmail SMTP (port 587, STARTTLS). Email contains:
- Run date + group
- Signal flips (most actionable — stocks that changed signal)
- Top green picks (score ≥ 7)
- Red watch list
- Average portfolio score
- Link to full GitHub Pages report

### `src/dashboard.py`

Writes two outputs to `public/data/`:
1. `manifest.json` — all tickers with summary fields (score, signal, price, archetype, sector, etc.)
2. `{TICKER}.json` — full per-ticker JSON including: card HTML (pre-rendered), 12-week history, Grok Logic panel, category scores, metrics, thesis, bull/bear flags

These files are committed back to the repo so Streamlit Cloud can read them.

### `src/database.py`

Maintains `data/history/{TICKER}.csv` — one row per pipeline run. Columns:
`date, score, signal, price, score_light, bull_count, bear_count`

Used by: Score Trend chart in Streamlit Home tab.

### `src/edgar.py`

Fetches from SEC EDGAR XBRL API when yfinance returns null fundamentals:
- `us-gaap/GrossProfit` + `us-gaap/Revenues` → gross margin
- `us-gaap/NetCashProvidedByUsedInOperatingActivities` − `us-gaap/PaymentsToAcquirePropertyPlantAndEquipment` → FCF

Useful for newer/smaller companies where yfinance data is spotty.

### `src/finnhub_client.py`

Free Finnhub API calls:
- `/stock/earnings` — last 4 quarters of EPS actual vs estimate → `fh_eps_surprise_pct`, `fh_beat`, `fh_consecutive_beats`
- `/stock/insider-transactions` — 90-day insider buy vs sell share count → `fh_insider_net_shares_90d`, `fh_insider_bullish`

### `src/macro.py`

FRED API series:
- `FEDFUNDS` — Federal Funds Rate
- `DGS10` — 10-Year Treasury Constant Maturity Rate
- `DGS2` — 2-Year Treasury (for yield curve: 10Y−2Y spread)
- `CPIAUCSL` — CPI Urban (13 months to compute YoY %)

Regime detection: "Easing" (Fed Funds falling), "Neutral", "Tightening" (rising rates).

### `src/telegram_bot.py`

Sends alerts via Telegram Bot API when:
- A ticker **flips signal** (e.g., from CONSOLIDATION to CONFLUENCE — actionable)
- A ticker hits a **strong entry** condition (score ≥ 7 + CONFLUENCE signal)
- **Weekly summary** — top 3 green picks with score + analyst upside

Setup: create a bot via BotFather, set `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` as secrets.

### `src/accuracy.py`

60-day backward-looking scorecard. For each past signal:
- Pulls price from 60 days ago and current price
- Checks if a BUY signal (CONFLUENCE on green score) beat SPY+2%
- Checks if a SELL signal (RISK WATCH on red score) avoided a drawdown
- Reports overall hit rate, signal-specific hit rates

Saved to `data/accuracy_report.json`, shown in the report's Accuracy Scorecard section.

### `src/thesis_ai.py`

Calls Claude Haiku (`claude-haiku-4-5-20251001`) with a structured prompt including:
- Overall score, factor scores
- Price, analyst upside, RSI
- Asks for exactly 2 sentences — no filler words

Returns `None` gracefully if `ANTHROPIC_API_KEY` is not set.

## 8. GitHub Actions Workflows

### `.github/workflows/watchlist.yml` — Main Pipeline

**Triggers:**
- `cron: "35 13 * * 1"` — Monday 6:35 AM PDT (13:35 UTC) → runs `--group owned`
- `cron: "35 13 * * 4"` — Thursday 6:35 AM PDT (13:35 UTC) → runs `--group watchlist`
- `workflow_dispatch` — manual trigger with group choice

**Two jobs:**

**Job 1: `analyse`**
1. Checkout repo (full history for clean push)
2. Restore signal cache from Actions cache
3. Set up Python 3.11
4. `pip install -r requirements.txt`
5. Determine group (Monday=owned, Thursday=watchlist, manual=input)
6. Run `python src/main.py --group $GROUP` with all API keys as env vars
7. Commit `data/history/`, `data/accuracy_report.json`, `public/data/` back to repo
8. Upload `public/` as Pages artifact
9. Save signal cache

**Job 2: `deploy`**
1. Deploy the uploaded Pages artifact to GitHub Pages
2. Outputs the live URL

**Why commit data back?** Streamlit Cloud reads `public/data/manifest.json` and `public/data/{TICKER}.json` from the repo. The pipeline generates these, so they must be committed to be visible to Streamlit.

### `.github/workflows/deploy-pages.yml` — Fast Static Redeploy

**Triggers:** Any push to `main` that changes files in `public/` or `src/`

This is a lightweight job (30 seconds) that just re-deploys whatever is in `public/` to GitHub Pages, without running the Python pipeline. Useful when you update `src/report.py` design and want the HTML change reflected immediately without waiting for the full pipeline.

### Theme

```toml
# .streamlit/config.toml
[theme]
base = "dark"
primaryColor = "#6366f1"        # indigo accent
backgroundColor = "#0f172a"     # deep navy
secondaryBackgroundColor = "#1e293b"  # card background
textColor = "#f1f5f9"
```

### Key Design Choices

- **Color only carries meaning** — green = strong, yellow = watch, red = caution. No decorative colors.
- **Uniform card backgrounds** — all cards are `#1e293b`. Color appears only as left-border accents.
- **Left-border accent pattern** — `border-left: 3px solid {color}` on dark `#1e293b` bg. Used everywhere.
- **`_sf()` safe-float helper** — all numeric values go through this, returns `None` for NaN/Inf/None instead of crashing.

### Per-Ticker Card Structure

```
Card (dark, border-left colored by conviction)
├── Header
│   ├── Ticker, Name, Archetype tag, Sector
│   ├── Signal badge (CONFLUENCE / CONSOLIDATION / SQUEEZE ON / RISK WATCH)
│   ├── Flip badge (↑/↓ if signal changed)
│   ├── Score trend (↑↓→)
│   ├── Claude score card (0–10, conviction level, signal text)
│   └── Grok score card (Grok Logic weighted, horizon, confidence %)
├── Dark Body

│       └── Right: Key Signals (bull/bear flags) + Score Sparkline (12-week trend)
```

### SVG Gauge

Pure Python/SVG — no JavaScript. Key implementation:
- Semicircle: center (75, 78), radius 55
- Background drawn as two 90° quarter-arcs (avoids SVG 180° ambiguity)
- Colored zone bands: red (0–4.5), yellow (4.5–7.0), green (7.0–10.0)
- Active filled arc: 180° → score_angle, using sweep=0 (CCW in SVG y-down = traces upper half)
- Needle: line from center to (cx + r*cos(angle), cy - r*sin(angle))
- Score number + "/ 10" label below center

### SVG Radar

6-factor hexagonal spider chart, pure SVG polygon:
- Factors: Fundamentals, Valuation, Technical, Moat, Macro, Sentiment
- Grid polygons at 25%, 50%, 75%, 100% of max radius (55px)
- Score polygon filled with `rgba(99,102,241,0.22)` (indigo)
- Each vertex labeled and color-coded green/yellow/red

## 11. Scoring System Deep Dive

### 5-Factor Live Score (Streamlit Analyze tab)

When you type a ticker in Analyze, a quick 5-factor score is computed live from yfinance:

| Factor | Key Metrics | Weight |
|--------|------------|--------|
| Valuation | Forward PE, analyst upside, recommendation | 25% |
| Growth | Revenue growth YoY, EPS growth | 20% |
| Margins | Gross margin, FCF margin, operating margin | 20% |
| Analyst | Recommendation mean, # of analysts, target upside | 20% |
| Technical | RSI, vs SMA-50, vs SMA-200, 52-week position | 15% |

### 11-Category Pipeline Score (GitHub Pages, weekly pipeline)

The full pipeline uses `thesis_scores.yaml` conviction scores + live data. Categories and what drives them:

**Fundamentals (20% weight)**
- Revenue growth YoY (archetype-scaled threshold)
- Gross margin, operating margin, FCF margin
- FCF absolute (positive = self-funding)
- Debt-to-equity ratio

**Valuation (15% weight)**
- Forward PE vs sector
- PEG ratio
- Analyst price target vs current price (upside %)
- EV/EBITDA

**Technical (15% weight)**
- RSI-14: <30 oversold, 30–70 neutral, >70 overbought, >80 extended
- Price vs SMA-50 and SMA-200
- 52-week range position
- Volume trend

**Sentiment (10% weight)**
- Analyst recommendation mean
- Short float % (inverse — lower is better)
- News sentiment score (if NEWS_API_KEY set)

**Management (8% weight)**
- Insider buying/selling (90-day net shares from Finnhub)
- EPS beat streak (from Finnhub)
- Guidance track record

**Moat (8% weight)**
- Gross margin stability (proxy for pricing power)
- Revenue retention / growth consistency
- Manual score from thesis_scores.yaml overrides

**Macro (7% weight)**
- Yield curve regime (FRED 10Y−2Y spread)
- Sector tailwinds/headwinds vs Fed policy
- Manual score from thesis_scores.yaml

**Catalysts (7% weight)**
- Next earnings date proximity
- Product cycle timing
- Manual score from thesis_scores.yaml

**Risk (5% weight)**
- Short float % (high = squeeze risk or bad sentiment)
- D/E ratio
- Drawdown from 52-week high

**Partnerships (3% weight)**
- Primarily manual score from thesis_scores.yaml

**Institutional (2% weight)**
- Institutional ownership % (>70% = smart money backing)
- Recent 13-F filing trends

### Grok Logic Panel (Second Opinion)

A simplified 6-bucket re-weighting of the same category scores, inspired by "what would a quant hedge fund think":

| Bucket | Categories Averaged | Grok Weight |
|--------|-------------------|-------------|
| Fundamentals | fundamentals | 30% |
| Valuation | valuation | 20% |
| Technical | technical | 20% |
| Moat + Catalysts + Management | average of 3 | 10% |
| Macro + Risk | average of 2 | 10% |
| Sentiment + Institutional | average of 2 | 10% |

Grok signal thresholds: ≥7.5 + fundamentals≥8 → "BUY on Dip" / ≥7.5 → "Strong Hold" / ≥6.5 → "Hold/Watch" / <5.5 → "Reduce/Sell"

## 13. Maintenance & Common Tasks

### Add a new ticker to your portfolio

1. Edit `config/owned.txt` — add `TICKER` (and optional archetype)
2. Optionally add entry to `config/thesis_scores.yaml` with your conviction scores
3. Push to GitHub
4. The ticker appears in the next Monday pipeline run (or trigger manually)

### Remove a ticker

1. Comment out or delete from `config/owned.txt` or `config/watchlist.txt`
2. Push. Gone from next run. History CSVs are kept for reference.

### Update your conviction scores (monthly recommended)

1. Edit `config/thesis_scores.yaml`
2. Update `last_updated` date
3. Adjust any category scores (0–10)
4. Push + trigger manual pipeline run to see updated report

### Trigger a manual pipeline run

GitHub → **Actions** → **Weekly watchlist digest** → **Run workflow** → select group → **Run workflow**

### View pipeline logs

GitHub → **Actions** → click the latest run → click the `analyse` job → expand each step

### Change the schedule

Edit `.github/workflows/watchlist.yml`:
```yaml
schedule:
  - cron: "35 13 * * 1"   # Monday 13:35 UTC
  - cron: "35 13 * * 4"   # Thursday 13:35 UTC
```
Change the cron expression. UTC time = your local time + UTC offset.
PST (winter) = UTC-8. PDT (summer) = UTC-7.

### Add a Telegram alert

1. Open Telegram → search `BotFather` → `/newbot` → follow prompts → copy token
2. Start a chat with your new bot
3. Visit `https://api.telegram.org/bot{YOUR_TOKEN}/getUpdates` → find `chat.id`
4. Add to GitHub Secrets: `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID`
5. Alerts fire automatically on next run with signal flips

## 14. Troubleshooting

### "Streamlit tabs are all broken / KeyError"

Usually means `tab_home` crashed on the Score Trend chart. Possible causes:
- `data/history/` has CSVs but they're missing the `score` column (old format)
- A CSV file is corrupted
- Fix: the code reads `data/history/{TICKER}.csv` as per-ticker files; it derives the ticker from the filename stem. If you see this, check that the CSV columns are: `date, score, signal, price, score_light, bull_count, bear_count`

### "% change showing as ±350%" in Analyze tab

`regularMarketChange` is a dollar change, not a percentage. Fixed code uses `regularMarketChangePercent` (decimal, e.g. 0.012 = 1.2%) and falls back to computing `(price − prevClose) / prevClose` directly.

### "Email not sending"

1. Check `GMAIL_USER`, `GMAIL_APP_PASSWORD`, `NOTIFY_EMAIL` secrets are set
2. Gmail App Password must be a 16-character code from Google Account → Security → App Passwords. Your regular login password will NOT work.
3. Check the Actions log for SMTP errors

### "GitHub Pages not updating"

1. Check that `deploy-pages.yml` ran (triggered on every push to main)
2. Check Settings → Pages → Source is set to "GitHub Actions" (not "branch")
3. Hard refresh the browser (Cmd+Shift+R or Ctrl+F5)

### "Streamlit not seeing pipeline data"

1. The pipeline must commit `public/data/` back to the repo (check the "Commit history data" step in Actions logs)
2. Streamlit reads from the repo at deploy time + caches for 1h. Clear cache: `st.cache_data.clear()` or just wait
3. Check that `public/data/manifest.json` exists in the repo

### "Finnhub data missing in Streamlit"

Streamlit Cloud secrets and GitHub Actions secrets are **separate**. Set `FINNHUB_API_KEY` in both places:
- GitHub: Settings → Secrets → Actions
- Streamlit: App settings → Secrets (TOML format)

### "SSL certificate error accessing Streamlit from work network"

Corporate networks (like PayPal's) intercept HTTPS. Solutions:
- Access from personal phone on mobile data
- In Chrome on work machine: type `thisisunsafe` on the error page (no click needed, just type)
- Use a personal laptop on home network

### "Pipeline runs but no history CSVs in repo"

The pipeline needs write permissions. Check: Repository → Settings → Actions → General → Workflow permissions → must be **"Read and write permissions"**.

## 15. Design Decisions & Why

### Why GitHub Actions instead of a server?

Zero cost, zero maintenance. GitHub Actions gives 2,000 free minutes/month for public repos. The pipeline takes ~6 minutes, twice a week = ~52 minutes/month. Well within free tier. No server to patch, monitor, or pay for.

### Why yfinance?

Free, no API key, covers 95% of what we need. EDGAR fills the gaps for fundamentals when yfinance returns null. Finnhub fills the gaps for insider data and EPS history.

### Why not use a broker API (Webull, Robinhood)?

Investigated Webull's unofficial Python library. The "Vega" AI analysis is app-only (server-side rendered, not accessible via API without authenticated device tokens). The unofficial library requires credentials stored in plaintext. Decision: skip broker integration, build richer narratives from Claude + Finnhub + EDGAR instead.

### Why two separate frontends (GitHub Pages + Streamlit)?

Different use cases:
- **GitHub Pages**: permanent URL in email, clean read-only HTML, best for "I got the email, let me check the report"
- **Streamlit**: interactive, live data, portfolio tracking, ad-hoc ticker analysis

They share the same data pipeline and stay in sync because Streamlit reads `public/data/*.json` which the pipeline generates.

### Why SVG charts instead of Plotly/Chart.js in the HTML report?

The GitHub Pages report is a **static** HTML file with no CDN dependencies. Plotly.js is ~3MB. Pure SVG charts are generated at report-build time by Python — zero runtime overhead, instant load, works offline. Streamlit (which has Plotly already installed) uses Plotly for the interactive animated gauge and radar.

### Why dark navy design?

Inspired by Webull's web UI — "makes you feel like you want to keep looking at the page." Key principle: **color only carries meaning**. Every colored element is green (strong), yellow (watch), or red (caution). No decorative colors, no rainbows. Uniform `#1e293b` card backgrounds everywhere; color appears only as left-border accents (`border-left: 3px solid {color}`).

### Why a 5-factor live score in Streamlit vs 11-category pipeline score?

The live score is computed in-browser from yfinance data in <1 second. The full 11-category pipeline score requires running the entire scorer (which takes ~30 seconds per ticker) plus thesis_scores.yaml context. The live score is deliberately simplified but still directionally correct for quick analysis. The pipeline's full score appears in Streamlit when you analyze a ticker that's in your watchlist (loaded from `public/data/{TICKER}.json`).

### Why separate schedules for owned vs watchlist?

Two reasons: (1) Spreads the API load — 39 tickers on Monday, 41 on Thursday. (2) Creates a natural rhythm — review portfolio positions at the start of the week, review new candidates mid-week. The Thursday run automatically skips any ticker that's already in owned.txt.
